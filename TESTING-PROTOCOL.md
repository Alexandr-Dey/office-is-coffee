# TESTING-PROTOCOL.md — Office is Coffee

> Протокол автоматизированного тестирования через Cowork (Claude in Chrome).
> Это **механические инструкции** для браузерного агента — без интерпретации, только точные шаги и проверки.
>
> Используй этот файл когда нужно протестировать сайт после изменений.
> Cowork должен следовать инструкциям буквально и выдавать структурированный отчёт.
>
> Версия: 1.0 (2026-04-07)

---

## 0. Инструкция для Cowork (читать первой)

### Кто ты в этом контексте

Ты — автоматизированный QA-инженер для проекта Office is Coffee. Твоя задача — **систематически проверять** что приложение работает после изменений кода.

### Общие правила

1. **Не интерпретируй** — выполняй инструкции буквально. Если шаг говорит "клик на кнопку 'Войти'", найди именно её и кликни.
2. **Не паникуй** при ошибках — фиксируй их и иди дальше. Твоя цель — собрать полную картину, а не остановиться на первой проблеме.
3. **Открывай DevTools Console** в начале каждого теста (F12 → Console). Записывай любые ошибки красным цветом — это критично.
4. **Делай скриншоты** при каждой неудаче (test failed).
5. **Используй формат отчёта** из раздела 8 — структурированный, с галочками.
6. **НЕ редактируй файлы в репозитории** — твоя задача только тестирование через браузер.
7. Если что-то требует логина и ты не залогинен — спроси пользователя.
8. Если что-то требует роли (бариста, CEO) — спроси пользователя как переключиться.

### Что НЕ тестировать (физические ограничения)

Эти тесты пропускай — они требуют реального устройства:
- ❌ iOS Safari в PWA standalone режиме
- ❌ Push-уведомления на iPhone
- ❌ Shake детекция (DeviceMotion)
- ❌ Реальная геолокация на улице
- ❌ "Красиво ли это выглядит" — субъективные оценки

### Что тестировать обязательно

- ✅ Все user flows (от регистрации до оплаты)
- ✅ Real-time обновления (статусы заказов)
- ✅ Корректность данных (цены, тексты, символы валюты)
- ✅ Работа форм (инпуты, селекты, кнопки)
- ✅ Навигация между страницами
- ✅ DevTools Console на наличие ошибок
- ✅ Network запросы к Firestore (в DevTools → Network)
- ✅ Адаптивность на разных ширинах окна

## 1. Подготовка к тестированию

### 1.1 Что нужно от пользователя перед стартом

Спроси у пользователя:

1. **URL для тестирования**: обычно https://office-is-coffee.vercel.app, но может быть preview URL
2. **Залогинен ли в Google аккаунт** в Chrome (нужен для авторизации в приложении)
3. **Какие роли тестируем**:
   - Только клиент?
   - Клиент + бариста?
   - Все три (клиент + бариста + CEO)?
4. **Какой раздел протокола запустить** (см. раздел 2):
   - Smoke test (5 минут — быстрая проверка что вообще работает)
   - Full regression (30 минут — полный прогон)
   - Targeted test (тестирование конкретной функции)

### 1.2 Подготовка окна браузера

1. Открой Chrome
2. Открой новое окно в режиме инкогнито (для чистого теста без кешей)
3. Установи размер окна **375×812** (имитация iPhone X) — F12 → Toggle device toolbar → выбери iPhone X
4. Открой DevTools (F12)
5. Перейди на вкладку Console — оставь открытой на протяжении всего теста
6. Перейди на вкладку Network → проверь что включена опция "Preserve log"

## 2. Виды тестирования (выбирай нужный)

### 2.1 SMOKE TEST (5 минут)
Быстрая проверка что **критические функции работают**. Используй после маленьких фиксов.

**Что входит**:
- Открытие главной
- Регистрация нового пользователя
- Просмотр меню
- Создание тестового заказа
- Console на ошибки

**Запускай**: раздел 3.

### 2.2 FULL REGRESSION (30 минут)
**Полный прогон всех тестов**. Используй после крупных изменений (новый блок задач P0/P1/P2 и т.д.) и перед запуском к тестерам.

**Что входит**:
- Все разделы 3-7

### 2.3 TARGETED TEST
Тестирование конкретной функции после её изменения.

**Какие есть**:
- Раздел 3: Авторизация и онбординг
- Раздел 4: Меню и заказ
- Раздел 5: Сцена кофейни
- Раздел 6: Админка баристы
- Раздел 7: CEO дашборд

## 4. Тесты: Меню и заказ

### TEST-MENU-001: Загрузка меню из Firestore

**Предусловие**: ты залогинен и на /menu.

**Шаги**:
1. Открой DevTools → Network
2. Очисти Network log
3. Перезагрузи страницу (Ctrl+R)
4. Найди в Network запросы к firestore.googleapis.com
5. Подожди полной загрузки страницы

**Ожидание**:
- В Network есть запрос к Firestore (collection menu_items)
- Меню отображается с **13 категориями**
- Видно минимум 60 позиций суммарно (если прокрутить все категории)

**Что проверить**:
- [ ] 13 категорий видны (Кофейная классика, Авторский, Айс, Какао, Домашний чай, Авторский чай, Матча, Айс ти, Молочные коктейли, Соки, Смузи фреш, Смузи молоко, Лимонады)
- [ ] Skeleton loading появлялся во время загрузки (проверь визуально или через slow network)
- [ ] Запрос к Firestore успешный (статус 200 или 101 для websocket)
- [ ] Нет хардкодного меню (если в Network НЕТ запроса к Firestore — это значит меню всё ещё хардкод, FAIL)

**Предусловие**: залогинен как клиент.

**Шаги**:
1. На /menu кликни на капучино
2. Выбери размер M, стандартное молоко

7. Выбери способ оплаты "Наличными"
8. Опционально: добавь комментарий "Тестовый заказ"
9. Нажми "Подтвердить заказ"
10. Подожди 3 секунды

**Ожидание**:
- Произошёл редирект на /order/[id]
- На странице видно статус заказа (изначально "new" или сразу "pending" если Cloud Function сработала)
- В DevTools → Network → запрос к Firestore с созданием документа в orders

**Что проверить**:
- [ ] Заказ создан (есть редирект на /order/[id])
- [ ] Статус виден на экране
- [ ] В Network виден запрос create к коллекции orders
- [ ] Cloud Function сработала: статус сменился на "pending" в течение 3 секунд

**Если статус остаётся "new" больше 5 секунд**: значит Cloud Functions не задеплоены или сломаны. Запиши как critical bug.

**Что проверить**:

- [ ] Real-time синхронизация с клиентом

**Дальше**:
5. Через минуту-две нажми "Готово"

## 9. Стандартные сценарии запуска

### Сценарий 1: После пуша Claude Code новых задач (например после P0)

```
1. Cowork: дождись 5 минут после git push (Vercel auto-deploy)
2. Открой URL продакшена
3. Запусти Smoke test (раздел 3 + ключевые тесты раздела 4)
4. Если smoke OK → запусти Full regression
5. Если smoke FAIL → останови, дай отчёт, ничего больше не делай
```

### Сценарий 2: Регулярная проверка перед демо тестерам

```
1. Cowork: запусти Full regression полностью
2. Дай полный отчёт
3. Если критических багов нет → можно показывать тестерам
4. Если есть → жди фиксов и повтори через час
```

## 11. Что делать если что-то пошло не так

### Сайт не открывается
- Проверь что URL правильный
- Проверь что сайт задеплоен (vercel.com/dashboard)
- Возможно идёт деплой — подожди 3 минуты

### Невозможно залогиниться через Google
- Проверь что Google провайдер включён в Firebase Console
- Проверь что в Firebase → Authentication → Settings → Authorized domains есть домен Vercel
- Возможно нужно подождать пару минут после включения провайдера

### Console полон ошибок Firebase
- Возможно `firestore.rules` не задеплоены
- Возможно env переменные не загружены в Vercel

### Тесты падают массово
- Не вини себя, вини конфигурацию
- Дай полный отчёт пользователю
- Не пытайся "починить" тестами — твоя задача найти, не исправлять

## 12. Безопасность

**Cowork НИКОГДА**:
- Не редактирует код в репозитории
- Не делает git коммитов
- Не запускает скрипты в терминале
- Не вводит пароли в формы (если только пользователь явно не разрешит)
- Не передаёт данные третьим лицам
- Не совершает реальные платежи

**Cowork ТОЛЬКО**:
- Тестирует через браузер
- Читает данные в DevTools
- Делает скриншоты
- Пишет отчёты
